1. EXECUTIVE SUMMARY
================================================================================

SIGNAL is an Android application that intercepts all incoming notifications,
classifies them using Groq AI, and enforces user decision-making on high-priority
signals. Unlike conventional notification managers or to-do apps, SIGNAL does not
passively organize information — it actively prevents important tasks from being
dismissed without conscious acknowledgment.

The core innovation is a Decision Enforcement Layer: a mandatory overlay that
requires the user to choose an action (Do Now / Schedule / Delegate / Ignore with
reason) before they can continue using their phone. This transforms smartphones
from passive information delivery devices into active behavioral guidance systems.

2. PROBLEM STATEMENT
================================================================================

2.1 The Modern Notification Paradox
-------------------------------------
Smartphones are the most powerful personal devices ever created. Yet despite
integrating communication, scheduling, work, and learning into a single system,
users consistently fail to act on important information they receive.

A typical user receives:
  - Messages from multiple platforms (WhatsApp, Gmail, Slack, Teams)
  - Deadline and assignment reminders
  - Meeting alerts and calendar notifications
  - Promotional and low-priority app updates

All of these arrive through a single, flat notification layer — with no
differentiation between a discount coupon and a critical assignment deadline.

2.2 The Core Structural Problem
---------------------------------
Current notification systems treat all information identically. A promotional
notification and a project deadline notification appear in the same format,
at the same priority level, with the same dismissal mechanism (swipe away).

This creates a fundamental issue: important signals are indistinguishable
from noise.

2.3 Human Behavioral Consequences
------------------------------------
From a behavioral science perspective:
  - Humans are not designed to process high-volume information streams
  - When overwhelmed, users delay decisions, ignore messages, rely on memory
  - This creates the pattern: Notification → Seen → Deferred → Forgotten

This is not a technology failure. It is a human-system interaction failure.

2.4 Why Existing Solutions Are Insufficient
---------------------------------------------
Notification Managers:
  - Organize or group notifications
  - Do NOT interpret meaning
  - Do NOT enforce action

Task Management Apps (Todoist, Notion, etc.):
  - Require manual entry
  - Depend entirely on user discipline
  - Are disconnected from real-time notification inputs

AI Assistants (Google Assistant, Siri):
  - Provide insights only when asked
  - Do not operate continuously in the background
  - Do not enforce decisions

KEY LIMITATION ACROSS ALL SYSTEMS: They assist users but do not change
user behavior.

2.5 The Real Gap
------------------
The missing layer in current systems is Decision Enforcement.

Today's systems:
  - Deliver information
  - Provide optional assistance

But they do NOT:
  - Ensure a user acknowledges important tasks
  - Ensure a decision is made
  - Ensure action follows

3. PROJECT OBJECTIVES
================================================================================

This project aims to build a system that:

  1. Identifies important signals within all incoming notifications
  2. Converts them into structured, actionable tasks using AI
  3. Forces the user to make an explicit decision (cannot be ignored)
  4. Tracks outcomes and enforces follow-through via scheduled re-surfacing
  5. Provides behavioral analytics to improve user accountability over time

Core Transformation:

  Raw Notification
        ↓
  AI Classification (Groq)
        ↓
  Structured Task with Context
        ↓
  Enforced Decision Overlay
        ↓
  Logged Action + Scheduled Follow-up
        ↓
  Dashboard Analytics + Accountability

4. SYSTEM PHILOSOPHY & BEHAVIORAL DESIGN
================================================================================

4.1 Three Core Principles
---------------------------

PRINCIPLE 1 — CONTEXT AWARENESS
  The system understands the meaning of incoming information, not just its
  surface content. AI extracts task, deadline, importance, and suggested
  actions from free-form notification text.

PRINCIPLE 2 — MANDATORY INTERACTION
  Users cannot passively dismiss high-priority notifications. They must
  respond with an explicit choice. This prevents the default human behavior
  of deferring decisions indefinitely.

PRINCIPLE 3 — BEHAVIORAL FEEDBACK
  All decisions are logged. The system tracks patterns: how often users
  defer, which tasks get ignored, what categories cause the most avoidance.
  This data is surfaced as a weekly accountability dashboard.

4.2 The Enforcement Principle
-------------------------------
Enforcement does NOT mean restriction. It means:
  - Preventing passive dismissal of important signals
  - Requiring conscious decision-making before moving on
  - Reintroducing unresolved tasks at scheduled intervals

This shifts the system from:
  Information Delivery → Behavioral Guidance

4.3 Behavioral Design Insight
--------------------------------
Users are significantly more likely to act when they are REQUIRED to decide,
compared to when they are merely allowed to ignore.

Instead of passive interaction (swipe away, read later), the system introduces:
  - Mandatory acknowledgment
  - Structured choice selection
  - Accountability tracking for all deferred or ignored tasks

6. CORE FEATURES & FUNCTIONAL REQUIREMENTS
================================================================================

--------------------------------------------------------------------------------
FEATURE 1: NOTIFICATION INTERCEPTOR SERVICE
--------------------------------------------------------------------------------
Description:
  A background Android service that silently captures all incoming notifications
  from all apps, extracts relevant metadata, and passes them to the AI pipeline.

Technical Component: NotificationListenerService

Functional Requirements:
  FR-1.1  Capture all incoming notifications system-wide
  FR-1.2  Extract: app package name, app display name, notification title,
          notification body text, timestamp, notification ID
  FR-1.3  Filter out: duplicate notifications, empty notifications, system UI
          notifications (e.g., charging, screenshot)
  FR-1.4  Pass captured data to GroqClassifier asynchronously (coroutine)
  FR-1.5  Run persistently in background; auto-restart if killed

Data Captured Per Notification:
  - sourceApp: String (e.g., "WhatsApp", "Gmail")
  - packageName: String (e.g., "com.whatsapp")
  - title: String
  - body: String
  - timestamp: Long (Unix epoch)
  - notificationId: Int

--------------------------------------------------------------------------------
FEATURE 2: GROQ AI CLASSIFICATION ENGINE
--------------------------------------------------------------------------------
Description:
  The intelligence core of the app. Each captured notification is sent to
  Groq API for real-time classification. Groq returns a structured JSON
  object containing importance level, category, extracted task, deadline,
  and suggested user actions.

Technical Component: GroqApiService.kt (Retrofit)

Functional Requirements:
  FR-2.1  Send notification data to Groq API endpoint
  FR-2.2  Use model: llama-3.3-70b-versatile
  FR-2.3  Receive and parse structured JSON response
  FR-2.4  Map response to ClassifiedTask data model
  FR-2.5  Handle API errors gracefully (fallback to manual review queue)
  FR-2.6  Response must be received within 2 seconds (Groq is very fast)
  FR-2.7  Store classified task in Room database

Groq API Endpoint:
  POST https://api.groq.com/openai/v1/chat/completions

System Prompt Template:
  "You are a notification classification AI. Analyze the notification below
  and return ONLY a valid JSON object with no markdown, no explanation.

  JSON schema:
  {
    importance: critical | high | medium | low,
    category: deadline | meeting | payment | message | reminder | promotional | other,
    task: string (extracted task in plain English),
    deadline: string | null (natural language, e.g. 'tonight 11:59 PM'),
    deadlineTimestamp: ISO8601 string | null,
    actions: string[] (2-4 suggested action labels),
    requiresEnforcement: boolean (true if importance is critical or high)
  }"

User Prompt Template:
  "App: {sourceApp}
   Title: {title}
   Message: {body}
   Received: {timestamp}"

Sample Response:
  {
    "importance": "critical",
    "category": "deadline",
    "task": "Submit IEEE paper by tonight 11:59 PM",
    "deadline": "tonight 11:59 PM",
    "deadlineTimestamp": "2024-01-15T23:59:00",
    "actions": ["Open WhatsApp", "Set Reminder", "Start Now", "Schedule"],
    "requiresEnforcement": true
  }

FEATURE 3: DECISION ENFORCEMENT OVERLAY
--------------------------------------------------------------------------------
Description:
  The centerpiece feature of SIGNAL. For all notifications where
  requiresEnforcement = true, a full-screen overlay is displayed. The user
  CANNOT dismiss this overlay by pressing Back or tapping outside. They must
  make an explicit choice from 4 options.

Technical Component: EnforcementOverlayActivity.kt

Functional Requirements:
  FR-3.1  Display as system-level overlay (TYPE_APPLICATION_OVERLAY)
  FR-3.2  Show: source app icon, notification title, AI-extracted task,
          detected deadline, importance badge, AI-suggested actions
  FR-3.3  Present 4 action buttons (cannot be skipped):
            → DO NOW: Marks task as in-progress, opens source app
            → SCHEDULE: Shows time/date picker, creates WorkManager reminder
            → DELEGATE: Opens share sheet to forward task to contact
            → IGNORE: Shows mandatory text field for reason (min 10 chars)
  FR-3.4  Back button press shows warning: "You must make a decision"
  FR-3.5  On any selection: save decision + timestamp to Room DB
  FR-3.6  Animate in from bottom with spring physics
  FR-3.7  Show countdown timer for time-sensitive tasks

Overlay Layout:
  [App Icon + Name]           [CRITICAL badge]
  ─────────────────────────────────────────
  Task: Submit IEEE paper
  Deadline: Tonight 11:59 PM  [⏱ 6h 23m]
  ─────────────────────────────────────────
  AI Suggested Actions:
  [Open WhatsApp] [Start Now]
  ─────────────────────────────────────────
  What do you want to do?

  [✅ DO NOW]     [📅 SCHEDULE]
  [👤 DELEGATE]  [❌ IGNORE]

  (If IGNORE selected)
  Why are you ignoring this?
  [________________________]  [Confirm]

--------------------------------------------------------------------------------
FEATURE 6: AUTOMATION ENGINE
--------------------------------------------------------------------------------
Description:
  Background automation that removes friction from acting on decisions.
  Handles rescheduling, re-surfacing, and calendar integration without
  requiring user to manually open apps.

Technical Components: WorkManager, Google Calendar API

Functional Requirements:
  FR-6.1  DEFERRED TASK RESURFACING
            → When user selects "Schedule" and picks a time
            → WorkManager schedules exact-time notification
            → At scheduled time: enforcement overlay reappears
            → If snoozed twice → escalation: cannot schedule again, must Do Now

  FR-6.2  OVERDUE ESCALATION
            → Tasks with deadline that have passed without completion
            → Re-appear in enforcement overlay with "OVERDUE" badge
            → Push notification with escalated urgency

  FR-6.3  GOOGLE CALENDAR AUTO-INTEGRATION
            → If AI detects category = "meeting" and a time is mentioned
            → Automatically create Google Calendar event
            → Prompt user to confirm (one tap)

  FR-6.4  DAILY DIGEST
            → Every day at 8 AM: push notification
            → "You have X pending tasks and Y overdue items"
            → Tap opens Task Board

  FR-6.5  SMART QUIET HOURS
            → User sets quiet hours (e.g., 11 PM – 7 AM)
            → During quiet hours: batch all enforcements
            → On wake: show consolidated enforcement for overnight tasks

7. TECHNICAL ARCHITECTURE
================================================================================

ARCHITECTURE PATTERN: MVVM + Clean Architecture

Layers:
  ┌─────────────────────────────────────────────┐
  │  UI LAYER (Jetpack Compose Screens)          │
  │  TaskBoardScreen, DashboardScreen,           │
  │  EnforcementOverlayActivity                  │
  └──────────────────┬──────────────────────────┘
                     │ observes StateFlow
  ┌──────────────────▼──────────────────────────┐
  │  VIEWMODEL LAYER                             │
  │  TaskViewModel, DashboardViewModel           │
  └──────────────────┬──────────────────────────┘
                     │ calls
  ┌──────────────────▼──────────────────────────┐
  │  REPOSITORY LAYER                            │
  │  TaskRepository                              │
  │  (single source of truth)                    │
  └────────┬─────────────────────────┬──────────┘
           │                         │
  ┌────────▼────────┐    ┌──────────▼──────────┐
  │  LOCAL DB        │    │  REMOTE API          │
  │  Room Database   │    │  Groq API Service    │
  │  TaskDao         │    │  (Retrofit)          │
  │  TaskEntity      │    │  Google Calendar API │
  └─────────────────┘    └─────────────────────┘
           ▲
  ┌────────┴─────────────────────────────────────┐
  │  BACKGROUND SERVICES                          │
  │  NotificationListenerService                  │
  │  WorkManager (TaskRescheduleWorker)           │
  └───────────────────────────────────────────────┘

Data Flow:
  Notification arrives
    → NotificationListenerService.onNotificationPosted()
    → GroqClassifier.classify(notificationData) [coroutine, IO dispatcher]
    → ClassifiedTask returned
    → TaskRepository.insertTask(classifiedTask)
    → If requiresEnforcement: Intent → EnforcementOverlayActivity
    → User makes decision → TaskRepository.updateDecision(taskId, decision)
    → If "Schedule": WorkManager.enqueue(TaskRescheduleWorker, scheduledTime)
    → TaskViewModel observes Room Flow → UI updates automatically

9. API SPECIFICATIONS — GROQ
================================================================================

Base URL     : https://api.groq.com/openai/v1
Endpoint     : POST /chat/completions
Auth Header  : Authorization: Bearer {GROQ_API_KEY}
Content-Type : application/json
Model        : llama-3.3-70b-versatile
Max Tokens   : 300
Temperature  : 0.1 (low for consistent JSON output)

REQUEST BODY:
{
  "model": "llama-3.3-70b-versatile",
  "max_tokens": 300,
  "temperature": 0.1,
  "messages": [
    {
      "role": "system",
      "content": "You are a notification classification AI. Analyze the notification and return ONLY a JSON object — no markdown, no explanation, no backticks. Use this exact schema: {importance: 'critical'|'high'|'medium'|'low', category: 'deadline'|'meeting'|'payment'|'message'|'reminder'|'promotional'|'other', task: string, deadline: string|null, deadlineTimestamp: ISO8601|null, actions: string[], requiresEnforcement: boolean}"
    },
    {
      "role": "user",
      "content": "App: {sourceApp}\nTitle: {title}\nMessage: {body}\nReceived: {formattedTimestamp}"
    }
  ]
}

RESPONSE PARSING:
  Extract: response.choices[0].message.content
  Parse as JSON → map to ClassifiedTask data class
  If JSON parse fails → fallback: importance=medium, requiresEnforcement=false

RATE LIMITS (Free Tier):
  Requests per minute  : 30
  Tokens per minute    : 14,400
  Daily token limit    : None (very generous)
  Cost                 : FREE

API KEY STORAGE:
  - Store in local.properties: GROQ_API_KEY=gsk_xxxxxxxxxxxx
  - Access via BuildConfig.GROQ_API_KEY
  - Never commit to version control
  - Add local.properties to .gitignore

10. DATA MODELS & DATABASE SCHEMA
================================================================================

--- DATA CLASS: NotificationData ---
data class NotificationData(
    val id: String,               // UUID generated on capture
    val sourceApp: String,        // "WhatsApp"
    val packageName: String,      // "com.whatsapp"
    val title: String,
    val body: String,
    val capturedAt: Long          // System.currentTimeMillis()
)

--- DATA CLASS: ClassifiedTask ---
data class ClassifiedTask(
    val notificationId: String,
    val importance: ImportanceLevel,  // CRITICAL, HIGH, MEDIUM, LOW
    val category: TaskCategory,       // DEADLINE, MEETING, PAYMENT, MESSAGE, etc.
    val task: String,
    val deadline: String?,
    val deadlineTimestamp: Long?,
    val suggestedActions: List<String>,
    val requiresEnforcement: Boolean
)

--- ROOM ENTITY: TaskEntity ---
@Entity(tableName = "tasks")
data class TaskEntity(
    @PrimaryKey val id: String,
    val sourceApp: String,
    val packageName: String,
    val originalTitle: String,
    val originalBody: String,
    val capturedAt: Long,
    val extractedTask: String,
    val importance: String,       // stored as string enum
    val category: String,
    val deadline: String?,
    val deadlineTimestamp: Long?,
    val status: String,           // PENDING, IN_PROGRESS, DONE, IGNORED
    val userDecision: String?,    // DO_NOW, SCHEDULE, DELEGATE, IGNORE
    val ignoreReason: String?,
    val scheduledFor: Long?,
    val decidedAt: Long?,
    val completedAt: Long?,
    val requiresEnforcement: Boolean,
    val isOverdue: Boolean
)

--- ROOM DAO: TaskDao ---
Key queries:
  @Query("SELECT * FROM tasks WHERE status = 'PENDING' ORDER BY deadlineTimestamp ASC")
  fun getPendingTasks(): Flow<List<TaskEntity>>

  @Query("SELECT * FROM tasks WHERE status = 'IGNORED' ORDER BY capturedAt DESC")
  fun getIgnoredTasks(): Flow<List<TaskEntity>>

  @Query("SELECT COUNT(*) FROM tasks WHERE capturedAt >= :startOfDay")
  suspend fun getTodayCount(startOfDay: Long): Int

  @Query("SELECT COUNT(*) FROM tasks WHERE userDecision = 'DO_NOW' AND capturedAt >= :startOfDay")
  suspend fun getTodayActionedCount(startOfDay: Long): Int

  @Query("SELECT * FROM tasks WHERE deadlineTimestamp < :now AND status != 'DONE'")
  fun getOverdueTasks(now: Long): Flow<List<TaskEntity>>

12. AUTOMATION ENGINE
================================================================================

12.1 WorkManager — TaskRescheduleWorker
-----------------------------------------
Triggered when user selects "Schedule" in enforcement overlay.

Worker Input:
  taskId: String
  scheduledTimeMs: Long

Worker Logic:
  1. Load task from Room by taskId
  2. Send local notification at scheduledTimeMs
  3. On notification tap → launch EnforcementOverlayActivity with taskId
  4. Track reschedule count in TaskEntity
  5. If rescheduleCount >= 2: in overlay, hide "Schedule" button,
     show message: "You've deferred this twice. Time to act."

WorkManager Constraints:
  .setRequiredNetworkType(NetworkType.NOT_REQUIRED)  // works offline
  .setBackoffCriteria(BackoffPolicy.LINEAR, 10, TimeUnit.MINUTES)

12.2 Overdue Escalation Worker
--------------------------------
Runs daily at midnight. Scans all tasks where:
  deadlineTimestamp < now AND status != DONE

For each: sets isOverdue = true in Room, sends escalation notification.

12.3 Google Calendar Integration
-----------------------------------
Trigger: AI classification returns category = "meeting" AND
         deadline or time detected in notification

Flow:
  1. Parse meeting time from deadlineTimestamp
  2. Call Google Calendar API: POST /calendars/primary/events
  3. Create event with title = extracted task, time = parsed time
  4. Show snackbar: "Meeting added to calendar ✓"
  5. If OAuth not granted: prompt once, then skip silently

12.4 Daily Digest (WorkManager — PeriodicWorkRequest)
-------------------------------------------------------
Frequency: Daily at 8:00 AM (user-configurable in settings)
Notification content: "Good morning! You have {x} pending tasks
and {y} overdue items waiting. Tap to review."
